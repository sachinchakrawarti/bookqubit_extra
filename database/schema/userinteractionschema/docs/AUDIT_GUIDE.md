# User Interaction Schema Audit Guide

## Overview

The **User Interaction Schema Audit Guide** defines how user interaction events are recorded, monitored, and traced throughout the BookQubit platform.

An audit system provides a permanent record of important actions, helping with security, troubleshooting, compliance, analytics, and operational monitoring.

Unlike activity logs, which are user-facing and may represent application events, **audit logs are immutable system records** intended for administrators and internal services.

---

# Objectives

The audit system is designed to:

- Record important database changes
- Track user actions
- Improve security
- Support compliance requirements
- Detect suspicious activities
- Simplify debugging and investigations
- Maintain historical records

# Audit Information

Each audit record should capture:

| Field | Description |
|--------|-------------|
| audit_id | Unique audit identifier |
| user_id | User performing the action |
| resource | Affected table or resource |
| resource_id | Record identifier |
| action | Create, Update, Delete, etc. |
| old_value | Previous data (optional) |
| new_value | Updated data (optional) |
| ip_address | Client IP address |
| user_agent | Client device/browser |
| timestamp | Event time |
| status | Success or failure |

# Suggested Audit Table

```sql
CREATE TABLE audit_logs (
    audit_id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    resource TEXT NOT NULL,
    resource_id INTEGER,
    action TEXT NOT NULL,
    old_value TEXT,
    new_value TEXT,
    ip_address TEXT,
    user_agent TEXT,
    status TEXT,
    created_at TEXT NOT NULL
);
```

# Audit Logging Methods

Audit records may be generated by:

- Application services
- Database triggers
- Background jobs
- Administrative tools
- Import/export utilities

Application-level logging is recommended for richer contextual information.

---

# Trigger Example

```sql
CREATE TRIGGER trg_reviews_audit
AFTER INSERT ON user_reviews
FOR EACH ROW
BEGIN
    INSERT INTO audit_logs (
        user_id,
        resource,
        resource_id,
        action,
        created_at
    )
    VALUES (
        NEW.user_id,
        'user_reviews',
        NEW.review_id,
        'CREATE',
        CURRENT_TIMESTAMP
    );
END;
```

# Query Examples

Recent audit records:

```sql
SELECT *
FROM audit_logs
ORDER BY created_at DESC;
```

Audit history for a user:

```sql
SELECT *
FROM audit_logs
WHERE user_id = 15
ORDER BY created_at DESC;
```

Audit history for a resource:

```sql
SELECT *
FROM audit_logs
WHERE resource = 'user_reviews'
AND resource_id = 245;
```

|------|-----------|

---

# Security

Audit logs should:

- Be append-only whenever possible.
- Restrict access to authorized personnel.
- Protect sensitive information.
- Avoid storing passwords or secrets.
- Record failed authorization attempts where appropriate.

---

# Performance Considerations

- Insert audit records asynchronously when appropriate.
- Index frequently queried columns.
- Archive old records periodically.
- Partition audit data in large deployments.
- Compress archived logs if necessary.

# Monitoring

Regularly review audit logs for:

- Repeated failed actions
- Excessive updates
- Bulk deletions
- Unusual login activity
- Unexpected permission changes
- Large data exports

---

# Best Practices

- Audit all sensitive operations.
- Keep audit records immutable.
- Store timestamps in UTC.
- Include sufficient context for investigations.
- Protect audit logs from unauthorized modification.
- Regularly archive and back up audit data.
- Monitor audit logs for anomalies.
- Test audit logging as part of application testing.
